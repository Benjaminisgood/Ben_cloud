# Standards Evolution Loop

本文件定义 **PROJECT_STANDARDS** 自身如何持续演进，目标是让未来新增项目保持统一且符合最佳实践。

## 1. 触发条件

以下情况出现时，需要更新本目录标准文档：

1. 新项目搭建过程中出现重复性踩坑。
2. 某条规范与当前最佳实践冲突（安全、可维护性、可观测性）。
3. 至少两个子项目出现相同架构问题，说明标准约束不足。
4. 框架或依赖出现大版本升级，影响标准做法。

## 2. 更新流程

1. 先在项目内复现问题并形成最小案例。
2. 明确“可执行规则”而非泛泛建议。
3. 更新本目录文档：
   - `FASTAPI_ENGINEERING_STANDARD.md`
   - `FASTAPI_UNIFICATION_PROGRESS.md`
   - `PLAYBOOK.md`
   - `registry.yaml`（如涉及新增项目模板字段）
4. 在提交说明中给出：
   - 变更前风险
   - 变更后收益
   - 对未来新项目的影响范围

## 3. 质量目标

1. 新项目从 0 到可运行服务无需临时约定。
2. 所有新项目可在统一 CI 门禁下通过（迁移、测试、编译）。
3. 关键工程决策（目录、分层、迁移、鉴权）有唯一标准来源。
4. 标准文档可被直接执行，不依赖口头补充解释。
